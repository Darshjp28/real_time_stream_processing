```
┌─────────────────────────────────────────────────────────────────────┐
│                    Spark Structured Streaming                       │
│                                                                     │
│  data/input/           readStream                                   │
│  (CSV batches)  ──────► (watched dir)                               │
│                              │                                      │
│                   to_timestamp("event_time")                        │
│                   filter(nulls)                                     │
│                              │                                      │
│                   withWatermark("event_time", "1 min")   ← STATE 1  │
│                              │                                      │
│                   groupBy(                                          │
│                     window("event_time", "2 minutes"),  ← TUMBLING  │
│                     patient_id                                      │
│                   )                                                 │
│                   .agg(avg("heart_rate"), count, max, min)          │

│   Console sink         Console sink          CSV sink               │

└─────────────────────────────────────────────────────────────────────┘
```

## Written Explanation (Assignment Requirements)

### 1. Why did I choose the Tumbling Window?

I chose a **Tumbling 2-minute window** for Scenario B because the clinical goal is to detect **sustained** abnormal heart rate — not a single spike.

A tumbling window divides time into completely **non-overlapping**, fixed-size intervals. Every sensor reading falls into exactly one window. If a patient's average heart rate exceeds 100 bpm in **two consecutive tumbling windows**, that represents at least **4 continuous minutes** of elevated HR — which is clinically meaningful and not attributable to transient noise or a single bad reading.

A **sliding window** would be incorrect here because overlapping windows reuse the same data points across multiple window instances. A single spike would appear in several windows, giving the false impression of a sustained condition and generating false alerts. A **session window** does not apply because ICU monitors produce continuous, regular readings with no natural inactivity gap that could define a session boundary.

The tumbling window also aligns directly with the assignment requirement: *"Flag patients exceeding 100 bpm in two consecutive windows."* The word "consecutive" implies distinct, non-overlapping time periods — exactly what a tumbling window provides.

---

### 2. Where does the pipeline require state?

The pipeline requires state in **two distinct places**:

#### State Point 1 — Implicit state in window aggregation (`withWatermark` + `groupBy window`)

```python
parsed
  .withWatermark("event_time", "1 minute")
  .groupBy(F.window("event_time", "2 minutes"), F.col("patient_id"))
  .agg(F.avg("heart_rate").alias("avg_heart_rate"), ...)
```

Spark internally maintains a **state store** (a hash map) of partial aggregates — running sum and count of `heart_rate` — per patient per window. This state accumulates as new micro-batches arrive. It is only finalized and evicted after:

```
max_seen_event_time − watermark_delay > window_end
```

Without this state, Spark could not compute the correct average across multiple micro-batches that fall within the same 2-minute window.

#### State Point 2 — Explicit state in `flatMapGroupsWithState`

```python
def detect_sustained_alert(patient_id, rows, state: GroupState):
    prev_window_start, prev_elevated = state.get if state.exists else (None, False)

    for row in rows:
        this_elevated  = row.avg_heart_rate > 100
        sustained      = this_elevated and prev_elevated   # ← requires state
        state.update((str(row.window_start), this_elevated))
        yield (patient_id, ..., sustained)
```

To determine whether **two consecutive windows** are both elevated, the pipeline must remember the previous window's alert flag per patient across micro-batches. This is explicit user-managed state via `GroupState`. The state stores `(prev_window_start, prev_was_elevated)` per patient and is updated every time a new window result arrives. It is automatically evicted after 10 minutes of inactivity.

Without this explicit state, each micro-batch would be evaluated independently, with no knowledge of prior windows, making consecutive-window detection impossible.
